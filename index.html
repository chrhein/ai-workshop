Kom i gang: bygg ditt første arbeidsområde "[ditt-navn]OS"

Nybegynner · ingen forkunnskaper · 40–50 min

Requirements

  • Claude Et Claude Pro- eller Max-abonnement. Mobil-dispatch (Remote Control / Dispatch) er tilgjengelig på både Pro og Max.
  • OpenAI En ChatGPT-konto (Codex følger med Plus/Pro/Business/Edu/Enterprise; mobil-dispatch krever macOS-versjonen av Codex-appen).
  • GitHub Copilot Et Copilot Pro / Pro+ / Business / Enterprise-abonnement. Mobil-styring skjer via remote control av Copilot CLI fra GitHub.com / GitHub Mobile. Får du Copilot-setet fra en organisasjon, må en org-/enterprise-eier først sette policyen «Store local sessions in the Cloud» til «View and control» (ukonfigurert som standard).
  • Lokal modell En Apple Silicon-Mac med en lokal modell som svarer på http://localhost:8000 — sett den opp via 🖥️ Sett opp en modell lokalt først.

1) Velg og sett opp klienten

Claude

Last ned Claude-appen fra claude.ai/download, logg inn med og bruk Cowork til å koble appen til en mappe på maskinen. Dette blir ditt arbeiidsområde, din [navn]OS i neste steg. Vil du heller jobbe i terminalen, installer Claude Code: 

# Terminal-alternativ (valgfritt)
curl -fsSL https://claude.ai/install.sh | bash
claude     # «Log in with Claude account», Pro/Max, ingen API-nøkkel
OpenAI · Codex

Last ned Codex-skrivebordsappen for macOS fra chatgpt.com/codex og logg inn med ChatGPT-kontoen din («Sign in with ChatGPT», ingen API-nøkkel). Dette blir klienten din, og den lar deg styre Codex fra ChatGPT-mobilappen senere. Vil du heller jobbe i terminalen, installer Codex CLI: 

# Terminal-alternativ (valgfritt)
curl -fsSL https://chatgpt.com/codex/install.sh | sh
# eller: npm install -g @openai/codex

codex      # «Sign in with ChatGPT», ikke API-nøkkel
GitHub Copilot

Installer GitHub Copilot-appen (desktop, macOS/Windows/Linux — teknisk preview) eller Copilot CLI, og logg inn med GitHub-kontoen din:

npm install -g @github/copilot
copilot    # naviger først til [navn]OS-mappa; godkjenn at du stoler på filene

Copilot kjører hver agent-økt i sin egen git-worktree — fint når du vil la flere oppgaver gå parallelt.

Lokal modell

Sjekk at modellen din svarer på http://localhost:8000 (se 🖥️ Sett opp en modell lokalt), og installer OpenCode som terminal-klient:

npm install -g opencode-ai
# Pek OpenCode på den lokale modellen i opencode.json:
#   "provider": { "lokal": { "npm": "@ai-sdk/openai-compatible",
#     "options": { "baseURL": "http://localhost:8000/v1" },
#     "models": { "mlx-community/Qwen3.6-35B-A3B-4bit": {} } } }
opencode   # start i [navn]OS-mappa

baseURL må ende på /v1. API-nøkkelen kan være hva som helst (f.eks. local).

2) Bygg arbeidsområdet ditt: [navn]OS · 2 min

Assistenten trenger en mappe å bo i. Lag den i Finder, eller bare be agenten gjøre det:

"Lag en mappe som heter hakonOS i Documents, med undermappene
 meetings/raw, meetings/summaries, daily-notes og tasks.
 Lag også en tom tasks/active.md."

Strukturen din ser nå slik ut:

hakonOS/
  meetings/
    raw/          (dump møtenotater her gjennom dagen)
    summaries/    (agenten lager rene oppsummeringer)
  daily-notes/    (én fil per dag)
  tasks/
    active.md     (alle aktive oppgaver)

Pek klienten din mot denne mappa (koble til mappa i appen, eller cd inn i hakonOS og start agenten der).

3) Gjør assistenten til din · persona + brukerkontekst, før første arbeidsflyt

Gjør dette før du gir den en jobb: en agent som vet hvem du er og hvordan du vil ha det, gir brukbare svar fra første prompt. Lag en instruksjonsfil i [navn]OS som klienten leser automatisk. Fyll ut denne lille malen — den er din:

# Om meg (brukerkontekst)
- Navn / rolle:
- Hva jeg jobber med til daglig:
- Verktøy jeg bruker (kalender, repo, chat …):
- Tidssone og arbeidstid:

# Slik vil jeg at du jobber (persona + standing orders)
- Mål for assistenten: hva skal den hjelpe meg med?
- Tone: (kort/uformell/direkte …)   ·   Språk: norsk
- ALLTID: vis hva du gjør før du skriver/sletter filer; grupper
  action items som [HØY]/[MEDIUM]/[LAV].
- ALDRI: rør .env eller secrets/; gjett på datoer/frister.

Hvor fila skal ligge avhenger av klienten:

Claude

Lagre som CLAUDE.md i [navn]OS (eller lim inn som «Project instructions» i appen).

OpenAI · Codex

Lagre som AGENTS.md i [navn]OS.

GitHub Copilot

Lagre som AGENTS.md eller .github/copilot-instructions.md i [navn]OS.

Lokal modell

OpenCode leser AGENTS.md i [navn]OS.

Test den med en gang: «Hvem er jeg, og hvordan vil jeg at du skal svare?» — den bør gjengi malen din.

4) Koble på dispatcheren (nå den fra mobilen)

Poenget med en assistent er at den er der når du ikke sitter ved maskinen. Koble klienten til en kanal du allerede bruker, og verifiser at meldinger går begge veier.

Claude

Slå på Remote Control / Dispatch: start en økt på maskinen, åpne Claude-appen på mobil, og send en oppgave dit. Den kjører på maskinen din (ingenting flyttes til skyen). Tilgjengelig på Pro og Max (beta); krever at desktop-appen er åpen og maskinen våken.

OpenAI · Codex

Åpne Codex i ChatGPT-mobilappen (iOS/Android), koble til Codex-appen på Mac-en din, og send/godkjenn oppgaver fra telefonen. (Codex-appen finnes nå også til Windows.)

GitHub Copilot

Start en interaktiv økt med copilot --remote i [navn]OS-mappa (eller slå på midt i økten med /remote on). CLI-en viser en QR-kode og en økt-spesifikk lenke; åpne den i GitHub Mobile eller på GitHub.com. Derfra ser du output i sanntid, svarer på spørsmål, godkjenner/avviser permission-forespørsler og planer, og sender nye prompts — alt mens økten kjører videre på macen din (verktøy, shell og filer blir værende lokalt).

Krav: GitHub-repo Arbeidsmappa må være et git-repo som er hostet på GitHub.com, ellers får du «Remote session disabled: not in a GitHub repository». Gjør [navn]OS til et (privat) GitHub-repo først: git init && gh repo create.

Kun du, innlogget med samme GitHub-konto, når økten. Maskinen må være på nett; bruk /keep-alive så den ikke sovner. Når du gjenopptar en økt, må du legge på flagget igjen: copilot --resume=<session-id> --remote eller copilot --continue --remote (eller sett det som standard i config med "remoteSessions"). Remote control krever en interaktiv økt — ikke --prompt/-p.

Lokal modell

En lokal modell har ingen ekstern dispatcher — du når den der maskinen er. Det er prisen for at alt blir værende lokalt. (Vil du ha mobil-tilgang senere, er det Nivå 3.)

Verifiser: send «Hva ligger i tasks/active.md akkurat nå?» fra mobilen og sjekk at du får svaret tilbake.

5) Første arbeidsflyt — to forslag

Nå er rammeverket på plass. Her er to forslag til hva du kan bruke [navn]OS til. De er ikke fasit — velg ett, prøv det, og bytt det ut med noe fra din egen hverdag når du har taket på flyten.

Forslag 1: møte-intelligens

Problemet: etter en dag med møter sitter du med et dusin notatbunker og null oversikt. Løsningen: dump rånotater i meetings/raw/ gjennom dagen, ikke organisert, kun ren braindump. På slutten av dagen sier du:

"Behandle alle møtenotater i meetings/raw fra i dag:

1. Lag rene oppsummeringer i meetings/summaries
2. Trekk ut ALLE action items til tasks/active.md
3. Grupper etter prioritet: [HØY], [MEDIUM], [LAV]
4. Ta med hvem som eier hva og frister

Prioritering:
- HØY: blokkerer noen andre, eller hard frist denne uka
- MEDIUM: viktig, men fleksibel timing
- LAV: greit å gjøre, men ingen avhengighet akkurat nå"

Forslag 2: daglig OS med /morning og /evening

To faste rutiner på 2–3 min som gir deg klarhet for hele dagen. Du lagrer dem som kommando-/prompt-filer klienten kjenner igjen, og kaller dem med ett ord. Innholdet er det samme uansett leverandør — bare plasseringen varierer:

Claude 
.claude/commands/morning.md
.claude/commands/evening.md
OpenAI · Codex 
~/.codex/prompts/morning.md
~/.codex/prompts/evening.md

Merk: custom prompts er på vei ut til fordel for skills (~/.codex/skills/) — de virker fortsatt, men nye flyter kan like gjerne lages som skills (se Nivå 2).

GitHub Copilot 
.github/prompts/morning.prompt.md
.github/prompts/evening.prompt.md
Lokal modell 
.opencode/command/morning.md
.opencode/command/evening.md

Innhold i morning.md

# Morgenrutine

Lag dagens notat i daily-notes med filnavn YYYY-MM-DD.md.
Still meg disse spørsmålene ett om gangen, vent på svar:

1. Hvordan føler du deg i dag? (energi, humør, hva ligger på hjertet)
2. Hva er dine tre viktigste prioriteringer i dag?
3. Hva ville gjort dagen vellykket?

Lag så notatet med dato/ukedag som tittel, energi-sjekk,
de tre prioriteringene som avkrysningsliste, suksesskriteriet,
og en «Notater»-seksjon for tanker gjennom dagen.

Innhold i evening.md

# Kveldsrefleksjon

Les dagens notat fra daily-notes.
Still meg disse ett om gangen, vent på svar:

1. Hva ble gjort i dag? Hva ble ikke gjort?
2. Hva lærte du eller la merke til om deg selv?
3. Hva er du takknemlig for i dag?

Oppdater dagens notat med en «Kveldsrefleksjon»-seksjon:
kryss av fullførte prioriteringer, hva du lærte, takknemlighet,
og overførte oppgaver til i morgen. Spør om de skal legges
til i morgendagens prioriteringer.
Det er DINE maler Rediger dem så de matcher hvordan du tenker. Tren og vær kreativ med prosjekter, noter nye vaner. Etter noen dager har du notater som agenten kan analysere.

Prøv å lag noe selv

Din egen flyt Forslagene over er bare to mønstre. Det beste caset er ett du faktisk har bruk for. Velg noe lite og repeterende fra din egen hverdag, og bygg det på samme måte:

  1. Skriv én setning: «Når jeg sier X, skal assistenten gjøre Y og lagre Z i [navn]OS.»
  2. Bestem hvilke filer/mapper den leser fra og skriver til.
  3. Skriv to–tre regler den alltid skal følge (språk, format, hva den aldri rører).
  4. Kjør det manuelt et par ganger. Funker det, lagre det som en kommando-/promptfil du kan kalle med ett ord.
Idéer å starte fra Ukeslogg fra daily-notes, «hva glemte jeg denne uka?» fra tasks/active.md, en leseliste du dumper lenker i, eller en enkel dagbok-/treningslogg. Hold det innenfor [navn]OS så lenge du lærer.
Sikkerhet mens du lærer Start alltid i en slik testmappe, ikke i hjemmemappa eller et ekte prosjekt. La godkjenning stå på for alt som skriver filer eller kjører kommandoer, godkjenn ett steg om gangen til du stoler på flyten.

Få assistenten til å gjøre noe

Mellomnivå · fra svar til handling · bygger på Nivå 1

Fortsett i arbeidsområdet ditt

Om du ikke har et arbeidsområde fra før, bygg det nå (Nivå 1). Det er her assistenten bor, og det er her den gjør ting. Lag en mappe, og åpne AI-verktøyet ditt i den.

Forslag 1: værmelding → klesvalg 🧥

"Hent værmeldingen for Oslo i dag fra yr (api.met.no) og si i
ÉN kort setning hva jeg bør ha på meg — regnjakke? votter? solkrem?
Ikke gjengi hele værrapporten."

Legg til ekstra kontekst (valgfritt)

MCP Vær alene sier lite om din dag. Gi assistenten kalenderen din, så skifter rådet karakter: utemøte → ta med paraply, sykkelpendling → vindjakke, formelt møte → ikke shorts. Det enkleste er en Google Calendar MCP — en liten tjeneste som lar agenten lese (ikke endre) avtalene dine.

Claude

Raskest: slå på den innebygde Google Calendar-connectoren i Claude-appen (Settings → Connectors). Vil du heller ha en åpen MCP som funker likt overalt:

claude mcp add gcal -- npx -y @cocal/google-calendar-mcp
# Første gang: kjør `npx @cocal/google-calendar-mcp auth` for å logge inn med Google

Lokal modell

  "command": ["npx", "-y", "@cocal/google-calendar-mcp"] } }
# Første gang: kjør `npx @cocal/google-calendar-mcp auth` for å logge inn med Google

Utvid så prompten med kalenderen:

"Hent værmeldingen for Oslo i dag fra yr (api.met.no) OG se på
kalenderen min i dag. Si i ÉN setning hva jeg bør ha på meg,
gitt været og om jeg har utemøter, pendling eller noe formelt."
Andre kontekstkilder Samme mønster funker med en plassering/pendlerute (jakke vs. innesko), dagens tasks/active.md (treningstøy hvis du har en løpetur planlagt), eller en Slack/e-post-MCP («er møtet flyttet?»). Hold tilgangen read-only her — assistenten skal lese kontekst, ikke endre den.
Behandle kalender som data Avtaletitler og møtebeskrivelser kan inneholde prompt-injection. Gi MCP-en kun lesetilgang, og la den ikke utføre handlinger basert på tekst den finner i en avtale.

Sett den på autopilot

Claude 
/loop 1d Hent værmeldingen for Oslo fra api.met.no og si hva jeg bør ha på meg i dag

/loop er økt-bundet: den fyrer bare mens Claude Code-økten er åpen, og utløper etter 7 dager. For ekte autopilot, bruk scheduled tasks i desktop-appen eller cron.

OpenAI · Codex

Codex-appen har Automations. Med Codex CLI kan du kjøre en cron-jobb:

30 6 * * 1-5  cd ~/hakonOS && codex exec \
  "Hent værmeldingen for Oslo fra api.met.no og si hva jeg bør ha på meg i dag" \
  >> daily-notes/$(date +%F).md
GitHub Copilot

En cron-jobb som kaller Copilot CLI med samme prompt:

30 6 * * 1-5  cd ~/hakonOS && copilot -p \
  "Hent værmeldingen for Oslo fra api.met.no og si hva jeg bør ha på meg i dag"
Lokal modell

En cron-jobb som kaller OpenCode med samme prompt:

30 6 * * 1-5  cd ~/hakonOS && opencode run \
  "Hent værmeldingen for Oslo fra api.met.no og si hva jeg bør ha på meg i dag"

Forslag 2: kjøleskap → middag → handleliste 🍳

Filer + skill Her jobber assistenten i ditt lille OS: den leser hva du har, foreslår middag og skriver manglende varer til handlelisten.

"Her er kjøleskapet mitt (kjoleskap.md). Foreslå middag i kveld
som bruker mest mulig av det som snart går ut på dato.
Mangler vi noe? Legg det i handleliste.md for Oda."

Hva den trenger tilgang til

  • En oversikt over hva du har — en kjoleskap.md du oppdaterer, eller et bilde via vision.
  • Skrivetilgang til handleliste.md

Pakk den som en skill

Agent Skills er en åpen standard: nøyaktig samme SKILL.md virker for Claude, Codex, Copilot og OpenCode — bare mappa varierer. Agenten leser description og laster skillen selv når den er relevant.

---
name: middagsplan
description: Bruk når brukeren ber om middag fra kjøleskapet eller handleliste.
---
Les kjoleskap.md og handleliste.md.
Foreslå én middag som bruker varer som snart går ut.
Skriv manglende varer til handleliste.md.
Ikke foreslå mer enn 30 min aktiv matlaging på hverdager.

.claude/skills/middagsplan/SKILL.md

Forslag 3: bestill ukeshandelen fra Oda 🛒

Browser + godkjenning Oda har ingen åpen MCP, så assistenten styrer Chrome i stedet. Dette er casen som viser at agenten kan gjøre en ekte oppgave i et ekte webverktøy. Dette er det første caset som trenger en nettleser, så vi kobler den opp her.

Først: koble Chrome til assistenten

Oda-casen bruker Playwright MCP. Det gir agenten en egen nettleser den kan åpne, klikke i og lese fra, uten at du deler din personlige nettleserdata.

  1. Lag en egen browserprofil for workshopen, f.eks. /Users/DITT_NAVN/ai-chrome-profile på Macen. Ikke bruk primærprofilen din.
  2. Koble Playwright MCP til agenten din:
Claude 
claude mcp add playwright -- npx -y @playwright/mcp@latest \
  --browser chrome \
  --user-data-dir /Users/DITT_NAVN/ai-chrome-profile

I ~/.codex/config.toml:

[mcp_servers.playwright]
command = "npx"
args = ["-y", "@playwright/mcp@latest", "--browser", "chrome",
        "--user-data-dir", "/Users/DITT_NAVN/ai-chrome-profile"]
GitHub Copilot

{ "mcpServers": { "playwright": {
  "command": "npx",
  "args": ["-y", "@playwright/mcp@latest", "--browser", "chrome",
           "--user-data-dir", "/Users/DITT_NAVN/ai-chrome-profile"] } } }
Lokal modell

  "command": ["npx", "-y", "@playwright/mcp@latest", "--browser", "chrome",

  1. Start agenten på nytt og spør: «Åpne oda.no i Chrome.»
  2. Logg inn på Oda manuelt i Chrome-vinduet agenten åpner. Ikke gi passordet til agenten.
  3. Test med noe ufarlig: «Søk opp melk på oda.no og les opp de tre øverste treffene.»
Browser-regel Assistenten kan klikke feil. Hold godkjenning på for alt som kjøper, sender, sletter eller endrer data. Ved betaling stopper agenten; du betaler selv.

Happy path

  1. Du holder en handleliste.md oppdatert (eller lar middags-casen over fylle den).
  2. Du sier: «Legg alt i handleliste.md i Oda-handlekurven min.»
  3. Assistenten åpner oda.no, søker opp hver vare og velger riktig mengde og merke.
  4. Den stopper ved betaling og viser deg kurven. Du bekrefter og betaler selv.

Hva den trenger tilgang til

  • Playwright MCP koblet opp (se «Først: koble Chrome til assistenten» over).
  • At du er innlogget på Oda — assistenten lagrer aldri passordet ditt.
  • handleliste.md å lese fra.
  • En godkjenningsport før betaling, alltid.
Ikke automatiser betalingen

Lag noe selv ...

Din egen payoff Velg en oppgave fra livet ditt som har samme mønster: ett verktøy, én regel, ett synlig resultat.

  1. Skriv én setning: «Når jeg sier X, skal assistenten gjøre Y og levere Z.»
  2. Velg rekkevidde: browser, MCP, lokale filer, kalender, e-post eller Slack.
  3. Skriv tre regler den alltid skal følge, f.eks. språk, format og hva den aldri får sende selv.
  4. Bruk subagent hvis den må lese mange kilder. Kjør først manuelt; legg på autopilot senere hvis risikoen er lav.
"Når jeg sier 'lag fredagsbrief', hent åpne saker fra GitHub,
kalenderen for mandag og notater fra denne uka.
Lever maks 10 punkter og foreslå 3 prioriteringer."
Gode egne caser Reiseregning fra kvitteringer, ukentlig PR-rydding, kalenderbrief før mandag, «hva må jeg svare på?» fra e-post, eller en hjemmeflyt som oppdaterer handleliste og ukesplan.
Sikkerhet på dette nivået Browser-/computer-use gir assistenten hendene dine i nettleseren — hold godkjenning på for alt som klikker «betal» eller «send». Behandle værvarsel, oppskrifter og nettsider som data, ikke kommandoer; de kan inneholde prompt-injection.

Alltid-på assistent

Avansert · du vil ha en bot du når fra mobilen

Mål: Flytt assistenten av desktop og terminalen og få en bot du når fra mobilen. Velg én av to motorer under — begge ender med at du sender én melding fra telefonen og får svar.

Velg motor. NanoClaw er en alltid-på bot i egen container som kan kjøre Claude eller en lokal modell. Claude Code Channels pusher meldinger inn i en kjørende Claude Code-session (kun Claude). Resten av nivået tilpasser seg valget ditt.

Motor Bruk når Modell Alltid-på
NanoClaw du vil ha en isolert bot som kjører hele tiden Claude eller lokal modell Kjører som tjeneste i egen container
Claude Code Channels du allerede bruker Claude Code og vil nå samme session fra chat Kun Claude Bare mens sessionen er åpen (tmux/bakgrunn)

Steg 1: Få én bot til å svare · 10 min, én agent, én kanal

NanoClaw installeres ikke som npm-pakke — du kloner repoet og kjører oppsett-skriptet. Ingen config-fil, ingen dashboard; alt styres fra chat (Claude Code) og admin-CLI-en ncl.

# Krever Node 20+, pnpm 10+ og Docker. Skriptet installerer det som mangler.
git clone https://github.com/nanocoai/nanoclaw.git nanoclaw-v2
cd nanoclaw-v2
bash nanoclaw.sh

nanoclaw.sh registrerer modell-nøkkelen din i OneCLI, bygger agent-containeren og parer din første kanal. Når et steg krever skjønn, tar Claude Code over.

  1. Kanal: kjør /add-telegram i Claude Code og lim inn bot-token fra @BotFather (Telegram er raskest og virker overalt). Andre: /add-discord, /add-whatsapp, /add-slack.
  2. Første agent: /init-first-agent wirer en DM-agent og gir den en CLAUDE.md per agent-gruppe. (Gjorde du Nivå 1–2? Lim system-prompten din inn der.)
  3. Send en melding fra telefonen (default trigger @Andy). Du har nå en bot som svarer. Restart en gruppe med ncl groups restart --id <group-id>.
Første payoff Boten kjører som tjeneste i egen container og svarer selv om terminalen er lukket. Hvilken modell den bruker velger du i Steg 2.

Krav: Claude Code v2.1.80 eller nyere og innlogging via claude.ai-konto — Console-/API-nøkkel-auth støttes ikke. bun --version må fungere, og på Team/Enterprise må admin ha slått på Channels.

Telegram er raskest i workshop:

/plugin install telegram@claude-plugins-official
/reload-plugins
/telegram:configure <token-fra-BotFather>

Start Claude Code med Telegram-kanalen:

claude --channels plugin:telegram@claude-plugins-official

Par og lås ned:

# Send en melding til boten i Telegram. Den svarer med en pairing-kode.
/telegram:access pair <code>
/telegram:access policy allowlist
Første payoff Meldingen din kommer inn i den kjørende Claude Code-sessionen som et channel-event, og Claude kan svare tilbake i Telegram. Sessionen må være åpen; bruk persistent terminal, tmux, screen eller bakgrunnsprosess for alltid-på.

Vil du teste uten ekte Telegram først, bruk demo-kanalen:

/plugin install fakechat@claude-plugins-official
claude --channels plugin:fakechat@claude-plugins-official
# åpne http://localhost:8787 og send en testmelding

Steg 2: Koble modell og runtime

NanoClaw kjører Claude som standard. Vil du holde alt på egen maskin, bytter du provideren per agent-gruppe til en lokal modell.

Med Claude (standard)

Ingenting ekstra å gjøre — nanoclaw.sh registrerte allerede modell-nøkkelen din i OneCLI i Steg 1. Boten kjører Claude. Hopp videre til Steg 3.

Med lokal modell

Antakelse: du har en OpenAI-kompatibel LLM-server som svarer på http://localhost:8000 (sett den opp via 🖥️ Sett opp en modell lokalt).

curl http://localhost:8000/v1/models     # bekreft server + finn modellnavnet

Installer OpenCode-provideren i Claude Code:

/add-opencode

Pek OpenCode mot endepunktet ditt (baseURL: http://localhost:8000/v1, model = det /v1/models returnerer). Provider settes per agent-gruppe — ncl groups config update --id <group-id> ..., eller bare be agenten gjøre det (admin-godkjenning kreves).

💡 Native localhost-OpenAI-compat er fortsatt under arbeid i NanoClaw (issue #1984); OpenCode med egendefinert baseURL er den fungerende ruten i dag. Bytt modell ved å oppdatere gruppe-configen og kjøre ncl groups restart --id <group-id>.

Ingen nanoclaw.json og ingen egen modellblokk. Kanalen pusher meldinger inn i den Claude Code-sessionen du startet med --channels, så modellen er den du allerede er logget inn med. Gjenbruk Nivå 1/2-instruksjonene dine i repoet: CLAUDE.md, skills, subagents og MCP ligger der Claude Code allerede leser dem.

Steg 3: Lås boten FØR du gir den tilgang

Test gjerne en prompt-injection-streng i en uskyldig kanal.

Sikkerhet i NanoClaw
  • Kun parede/godkjente avsendere slipper inn (unknown_sender_policy); styr tilgang med ncl roles / ncl members, og se avviste meldinger med ncl dropped-messages list.
  • Krev mention i grupper via wiring-reglene (/manage-channels).
  • Krev godkjenning for handlinger som bruker legitimasjon: OneCLI håndterer hemmeligheter og godkjenningspolicy (web-UI på http://127.0.0.1:10254), og NanoClaw ruter godkjenningen til en admin. Self-mod (installere pakker / koble MCP) krever admin-godkjenning.
  • Hemmeligheter ligger i OneCLI-vaulten, aldri i env eller chat. .env brukes bare til bootstrap, men ikke commit dette.
  • Hver agent-gruppe kjører isolert i egen Docker-container og ser bare det du mounter.
Autovern før fart — Claude Code Channels
  • Bruk pairing + access policy allowlist. Alle som kan svare gjennom en kanal med permission relay kan også godkjenne eller avvise verktøybruk — allowlist derfor kun folk du stoler på.
  • Boten kjører i din lokale Claude Code-session; permission prompts kan stoppe jobben til du svarer.
  • API-nøkler/tokens i .env eller passordhåndterer, aldri i kode eller git.

Steg 4: Gi den ekte capabilities + planlagte oppgaver

Koble på verktøy (MCP) så den faktisk gjør noe, og flytt Nivå 2-briefen din inn som en recurring jobb.

Idé Verktøy-miks (MCP)
Daglig brief kl. 07:00 (kalender + e-post + nyheter) google-calendar + gmail + rss
Utviklerassistent i chatten github + terminal (med godkjenning)
Møtereferat-maskin zoom + whisper + notion

Koble på verktøy via MCP: ncl groups config add-mcp-server ... (eller be agenten, med admin-godkjenning). Planlagte oppgaver lager du ved å be boten i naturlig språk — «@Andy hver hverdag 07:00, send meg en brief» — så kjører den Claude på timeplan og melder deg tilbake.

Claude Channels er event-drevet: chat, CI, webhooks og varsler kan pushe inn i sessionen. MCP-servere og skills ligger der Claude Code allerede leser dem (.mcp.json, CLAUDE.md). For timer-baserte jobber bruker du Claude Code scheduled tasks.

Steg 5: Skaler til flere assistenter 🎯

Én assistent som skal kunne alt blir middelmådig på alt. Del den i spesialister — én for jobb, én for kode, én for hjem — hver med egen systemprompt og egne verktøy. Ferdige system-prompter ligger i mappen nivaa-3/agents/.

NanoClaw bruker agent-grupper Hver spesialist er sin egen agent-gruppe med egen CLAUDE.md, eget minne og egen container. Du ruter ved å wire ulike kanaler/chatter til ulike grupper — ikke ved å bytte ut en config-blokk. 
ncl groups create --name jobb      # kalender + e-post
ncl groups create --name dev       # github + CI/logg
ncl groups create --name hjemme    # handel + familie

# wire hver gruppe til sin kanal/chat (eller bruk den guidede /manage-channels)
ncl wirings create --messaging-group <slack-jobb>   --agent-group jobb
ncl wirings create --messaging-group <slack-dev>    --agent-group dev
ncl wirings create --messaging-group <slack-hjemme> --agent-group hjemme

Legg verktøy på hver gruppe med ncl groups config add-mcp-server. System-promptene i agents/ blir gruppenes CLAUDE.md. Meldinger fra dev-kanalen går til dev-gruppa, som svarer i samme tråd — hver spesialist i egen container, ett uhell sprer seg ikke. Vil du ha én felles inngang som ruter videre, kan dispatcher.md kjøre som sin egen gruppe foran de andre.

Claude Code bruker subagents som spesialister i samme session. Du ber Claude delegere («be dev-spesialisten fikse CI»), og hver subagent har egen systemprompt og eget verktøysett. System-promptene i agents/ blir subagent-definisjoner. Alt deler samme session og samme isolasjonsnivå som hovedøkten.

4

OpenClaw i en isolert VM

Ekspert ·

Modellstøtte: Inne i VM-et kjører OpenClaw gjennom abonomentet ditt eller API-nøkkeler. Tre spor virker her: Claude (API-nøkkel, eller gjenbruk av Claude Code-innlogging), OpenAI (Codex-abonnement via OAuth, eller OPENAI_API_KEY) og en lokal OpenAI-kompatibel modell.

Steg 1: Spinn opp VM-et

Default: Lume — Cua sitt CLI for native VM-er på Apple Silicon.

Krav (Lume) Apple Silicon (M1–M5), macOS 13+ og ~30 GB ledig disk.
# Installer Lume
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/trycua/cua/main/libs/lume/scripts/install.sh)"

# Lag og start en VM
lume create assistent-vm --os macos --ipsw latest
lume run assistent-vm

Steg 2: Installer OpenClaw inni VM-et

Alt herfra skjer i terminalen inne i VM-et, akkurat som Nivå 3 Steg 1, bare at hostenmaskinen ikke ser noe av det:

# Krever Node 24 (anbefalt) eller Node 22.19+
npm install -g openclaw@latest
openclaw onboard --install-daemon   # installerer Gateway-daemonen (launchd) + leier deg gjennom oppsett
openclaw gateway status             # Control UI: http://127.0.0.1:18789/ (inne i VM-et)

onboard kan sette opp både modell og auth for deg — vil du heller gjøre det for hånd, ser du de tre sporene under. All config ligger i ~/.openclaw/openclaw.json inne i VM-et; system-prompten din bor i workspace-en (~/.openclaw/workspace/SOUL.md / AGENTS.md).

Steg 3: Velg modell og auth (inne i VM-et)

Modellen velges som provider/modell-id under agents.defaults.model i ~/.openclaw/openclaw.json. Dette er den eneste delen som varierer per leverandør, men resten av OpenClaw (workspace, skills) er likt:

Claude
Historikk: abonnement-OAuth ble sperret april 2026 Anthropic stengte tredjeparts OAuth-bruk av Claude-abonnement 4. april 2026 (tredjepartsverktøy trekker nå fra «extra usage»).Sjekk OpenClaw docs om Claude Code er tilgengelig igjen.

To måter:

  • API-nøkkel: sett ANTHROPIC_API_KEY i VM-ets miljø og velg modell:
openclaw models set anthropic/claude-opus-4-6
openclaw models status
  • Claude CLI-gjenbruk: installer og logg inn i Claude Code inne i VM-et, behold canonical modell-ref og velg CLI-runtime:
{
  agents: {
    defaults: {
      model: { primary: "anthropic/claude-opus-4-7" },
      agentRuntime: { id: "claude-cli" }
    }
  }
}
OpenAI

Prefikset velger ruta: openai/gpt-5.5 = direkte API-nøkkel, openai-codex/gpt-5.5 = Codex-abonnement via OAuth. To måter å autentisere:

  • Codex-abonnement (OAuth): kjør openclaw onboard --auth-choice openai-codex (eller openclaw models auth login --provider openai-codex; legg på --device-code i headless VM). Ingen API-nøkkel — auth kommer fra ChatGPT/Codex-kontoen din, og OpenAI tillater eksplisitt abonnement-OAuth i verktøy som OpenClaw.
  • API-nøkkel: sett OPENAI_API_KEY i VM-ets miljø. Mest forutsigbart for en alltid-på gateway-host.

Sett modellen via CLI:

# Abonnement (OAuth):
openclaw models set openai-codex/gpt-5.5
# API-nøkkel:
openclaw models set openai/gpt-5.5
openclaw models status            # bekreft primary + auth

… eller direkte i ~/.openclaw/openclaw.json:

{
  agents: {
    defaults: {
      model: { primary: "openai-codex/gpt-5.5" }
    }
  }
}
Lokal modell

To måter å gi VM-et en lokal, OpenAI-kompatibel modell:

  • Ollama inne i VM-et (enklest, alt på ett sted): installer Ollama i VM-et, ollama pull <modell>, og bruk modell-ref ollama/<modell>.
  • oMLX på verten: la oMLX servere på http://<vert-IP>:8000/v1 og registrer den som egen provider. Bruk hostens IP — ikke localhost, som inne i VM-et peker på VM-et selv.

Egen provider mot oMLX i ~/.openclaw/openclaw.json:

{
  models: {
    providers: {
      omlx: {
        baseUrl: "http://<vert-IP>:8000/v1",
        apiKey: "local",
        api: "openai-completions",
        models: [
          { id: "mlx-community/Qwen3.6-35B-A3B-4bit",
            name: "Qwen 3.6 35B (lokal)" }
        ]
      }
    }
  },
  agents: {
    defaults: {
      model: { primary: "omlx/mlx-community/Qwen3.6-35B-A3B-4bit" }
    }
  }
}

Modell-ref-en splittes på første /: provider omlx + modell mlx-community/Qwen3.6-35B-A3B-4bit. apiKey kan være hva som helst. Verifiser med openclaw models status.

Steg 4: Lås det — og snapshot før første auth

VM-et er den ytre grensa, men autovernet fra Nivå 3 gjelder fortsatt inne i det:

Autovern i VM-et
  • DM-pairing står på som standard (dmPolicy="pairing"): ukjente avsendere får bare en pairing-kode. Godkjenn deg selv med openclaw pairing approve <kanal> <kode>.
  • Behandl innkommende DM-er og ekstern tekst som untrusted.
  • Noe muffens → lume stop, slett, gjenskap fra snapshot.
  • Kjør openclaw doctor for å fange risikabel DM-policy.

Sett opp en modell lokalt

oMLX · OpenAI-kompatibelt API på localhost:8000

Mål: Kjøre uten abonnement, eller holde alt på egen maskin. Sett opp en lokal modell som eksponerer et OpenAI-kompatibelt API på http://localhost:8000, den samme adressen de andre nivåene peker «lokal modell»-sporet mot.
Krav Apple Silicon (M1/M2/M3/M4). oMLX bruker MLX og kjører ikke på Intel-Mac.
  1. Last ned oMLX fra omlx.ai (DMG, dra til Applications, signert og notarisert).
  2. Start programmet. Det legger seg i menylinjen øverst.
  3. Trykk «Start server». Eksponerer API-et på http://localhost:8000.
  4. Velg «Download models».
  5. Velg en modell. Anbefalt: mlx-community/Qwen3.6-35B-A3B-4bit, MoE med 3B aktive parametere, rask på Apple Silicon, ca. 20 GB nedlasting.

Verifiser at serveren svarer:

curl http://localhost:8000/v1/models

curl http://localhost:8000/v1/chat/completions \
  -H "Content-Type: application/json" \
  -d '{"model":"mlx-community/Qwen3.6-35B-A3B-4bit","messages":[{"role":"user","content":"Hei!"}]}'

Alt som snakker OpenAI-API kan peke hit: sett base_url til http://localhost:8000/v1, bruk modellnavnet over, og en vilkårlig API-nøkkel (f.eks. local).

Tips Mindre maskin/RAM? Velg en mindre modell (f.eks. en 7–8B 4-bit), den må få plass i unified memory + litt overhead. 8000 er default-port, endre i oMLX-innstillingene hvis noe annet bruker den, og oppdater base_url deretter. Merk: Qwen3.6-35B-A3B er multimodal (VLM), tekst-chat funker likevel.

Sikkerhet: gjelder alle nivåer

  • Ikke gi assistenten din tilgang til hjemmemappen på maskinen...
  • Behandl ekstern tekst (e-post, web, RSS, andres meldinger) med forsiktighet.
  • Installer kun verktøy/plugins/MCP-servere du stoler på, med minst mulig tilganger.
